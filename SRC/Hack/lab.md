# Лабораторная работа по анализу уязвимостей программного обеспечения и разработке эксплойтов в среде эмуляции DOSBOX

# 1. Цель работы
Разработка программы авторизации пользователя с искусственно внедрёнными уязвимостями безопасности, проведение взаимного обмена программным обеспечением с партнером, анализ бинарного кода партнера и реализация методов эксплуатации выявленных уязвимостей для получения несанкционированного доступа.

# 2. Описание разработанной программы и внедренных уязвимостей

В ходу выполнения работы была написана программа авторизации, содержащая уязвимости двух классов:
1. Переполнение буфера в сегменте кода.
Буфер для хранения вводимого пароля размещен непосредственно в сегменте кода.
Механизм ввода строки не осуществляет проверку длинны вводимого пароля.
При выходе за пределы буфера строка перезаписывает память сегмента, включая стек программы.
Эксплуатация данной уязвимости позволяет изменить адрес возврата из процедуры ввода.
2. Исполнение кода из сегмента данных.
Функция хеширования при старте программы копируются с видеопамять (сегмент `0b800h`), откуда производится её дальнейшее исполнение.
Адрес функции определяется значением неинициализированной области памяти, что усложняется статистический и эмпирический анализ, но не предотвращает возможности подмены.
Уязвимость заключается в возможности удаления алгоритма хеширования, что позволяет проходить проверка хеша.

# Моя программа

В собственную программу были заложены следующие уязвимости:
- Буфер для вводимого пароля хранился прямо в сегменте кода после всех инструкций, длина вводимого пароля была неограниченна и процедура ввода пароля вызывалась из основного кода.
Для взлома такой уязвимости нужно было ввести пароль, такой длинны, что бы дойти до данных стека, и заменить тем самым адрес возврата из процедуры.
- Хеш-функция в начале программы копировалась в видеопамять и в дальнейшем исполнялась оттуда. Для взлома можно быть заменить её на свою функцию, которая возвращает необходимый хеш.
Стоит отметить, что для усложнения взлома, адрес хеш-функции задавался с помощью значения неинициализированной памяти, что обеспечивало защиту от простого подбора адреса.

# Анализ программы партнера

Проанализируем программу партнера.
В самом начале производится первоначальная инициализаций регистров
```
00000000: BB7201          mov bx,0172
00000003: B90000          mov cx,0000
00000006: BA9090          mov dx,9090
00000009: B80001          mov ax,0100
```
Особое внимание обратим на регистр `dx`. В дальнейшем его значение нам понадобиться.

Затем программа в цикле обрабатывает символы из стандартного ввода.
```
0000000C: CD21            int 21
0000000E: 3C0D            cmp al,0D
00000010: 7406      (1)   je 00000018
00000012: 8807            mov byte ptr [bx],al
00000014: 43              inc bx
00000015: 41              inc cx
00000016: EBF1      (2)   jmp 00000009
```
В цикле значение, считанное из 21 порта, записывается по адресу `ds:[bx]`, где `bx` изначально инициализирован значением `172h`. В программе по этому адресу располагается большой буфер памяти.
Если пользователь ввел символ с ASCII-кодом `0Dh`, который соответствует символу `\n`, то цикл завершается.

Заметим, что длина вводимой строки ничем не ограничена, поэтому возникает возможность переполнения внутреннего буфера.

Мы подошли к первой уязвимости.
```
00000018: 89161C01        mov word ptr [011C],dx
0000001C: 90              nop
0000001D: 90              nop
```
В стандартной ситуации регистр `dx` на протяжении предыдущей части программы не должен изменяться, поэтому в `[011C]` записалось бы `0909`, что соответствует инструкциям `nop`.
Однако, если во время обработки ввода регистр `dx` измениться, место `nop` будут выполняться другие инструкции.

Затем вызывается процедура хеширования
```
0000001E: BE7201          mov si,0172
00000021: E86301    (3)   call 00000187

00000187: 33DB            xor bx,bx
00000189: 33C0            xor ax,ax
0000018B: AC              lodsb al,ds:[si]
0000018C: 03D8            add bx,ax
0000018E: E2FB      (2)   loop 0000018B
00000190: C3              ret
```
В качестве хеш-функции используется простая сумма ASCII-кодов символов, коллизия которой высока и подобрать строку под нужный хеш не невозможно.
Это можно также рассматривать в качестве уязвимости.

После хеширования введенной строки происходит считывание необходимого хеша из памяти.
```
00000024: A1D601          mov ax,[01D6]
```

Важная деталь, адрес `[01D6]` идет сразу после внутреннего буфера для введенной строки, поэтому при переполнении хеш настоящего пароля может быть изменен.
Это и есть вторая уязвимость программы.

# Процесс взлома

Для использования первой уязвимости, связанной с записью значения регистра `dx`, необходимо во время чтения входной строки изменить его значение.
В самом цикле ни одни из инструкций не может ни каким образом затронуть его.
Поэтому для взлома была написана программа резидент, которая привязана к 1 прерывания (прерыванию клавиатуры) и при нажатии на клавишу `1` изменяет значение регистра `dx` на код обычного относительного `jump`-a.
Таким образом, вместо инструкций `nop` происходит выполнение короткого `jump`-a на необходимый адрес, который соответствует успешному входу.

Вторую уязвимость можно использовать несколькими способами:
- Подбор строки для нужного хеша
- Подмена стандартного хеша
- Подмена хеш-функции

Поподробнее разберем каждую возможность.

## Подбор хеша

Благодаря простой хеш функции и практически неограниченной длине строки подбор строки становится возможным и даже легким процессом.
Значение необходимого хеша можно взять прямо из программы - и это `0437h`.
Символы необходимой строки должны в сумме дать такое число.
Например, таковой является строка: `zzzzzzzzg` - 8 * `z` + `g` = `3d0h` + `67h` = `437h`

## Подмена стандартного хеша

При переполнении буфера мы можем затронуть реальный хеш, тем самым изменив значение, с которым будет сравниваться хеш входной строки.
Для этого хеш последних двух символов строки, который подменят собой необходимый хеш, должен быть равен хешу всей строки.
Для этого мы можем использовать строку из пробелов, т.к. их ASCII-код равен 0.

Что бы не вводить всю строку вручную, можно использовать перенаправление ввода на файл со строкой.

## Подмена хеш-функции

Хеш функция располагается ниже буфера, поэтому при переполнении мы можем изменить и её.
Однако для этого потребуется повлиять на большой кусок кода, лежащий между буфером и хеш-функцей, что в принципе реально, но не эффективно.

# Написание программ для взлома

Для взлома первой уязвимости была написана программа резидент, связанная с прерыванием клавиатура, которая при нажатии на `1` изменяет значение регистра `dx` на необходимое.
В данном случае это `EB18` - соответствующее короткому jump-у на `18h` байтов вперед (`jmp 18h`).
Тогда, при вводе пароля достаточно нажать на клавишу `1` и любой пароль подойдет.

Для взлома второй уязвимости можно было принято решение подмены стандартного хеша.
Для этого достаточно было бы ввести пароль длинной 102 пробела, что легко сделать с помощью перенаправления стандартного ввода, однако в процессе возникла проблема с перенаправлением.
Вместо файла пришлось писать резидента, завязанного на `21h` прерывание, которой при вызове с `ah = 01h` записывает в `al` `0h`.
Делает он так 102 раза, затем вместо `0h` вводит `0Dh` (символ новой строки `'\n'`) и больше никак не влияет на дальнейшую работу.
